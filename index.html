Key insight

Chat = you talk to Claude. Code = Claude acts in your codebase. Cowork = Claude acts on your desktop. All three share the same underlying model.

Skills and agents
The infrastructure layer underneath all three interfaces.
🛠️ Skills
Skills
SKILL.md files that give Claude a specific playbook — a PPT builder, brand guide, PRD writer, or anything you define. Loaded on-demand when a trigger phrase is detected.
Analogy: Specialist shops in the city. You walk in, they know exactly what to do.
🚴 Agents
Agents
Claude takes multi-step actions autonomously — web search, file edits, API calls, tool use. You describe the goal; Claude figures out the steps.
Analogy: Couriers. You say "deliver this", they figure out the route.
🔌 MCP tools
MCP connectors
Integrations that let Claude talk to external apps — Jira, Notion, Figma, GitHub, Slack and more. Connected once, available in every session.
Analogy: Phone lines into the city. The courier can call any department.
How skills are loaded per interface
The mechanism varies, but the principle is the same — context shapes what Claude does.
Chat
Reads SKILL.md from /mnt/skills/ on first trigger in a session. Skills are scoped to your account — shared across all chats.
Claude Code
A CLAUDE.md file in your repo root sets persistent project context — coding standards, architecture notes, key patterns. Read every session automatically.
Cowork
Workflow automations define the steps. Claude executes them across your desktop apps — no terminal or coding required.
Memory
Summaries from past chats are stored and injected into future sessions. Claude remembers your name, role, tools, and preferences across conversations.
The Git railway
Ten stops from remote repo → your machine → back to GitHub → merged to main. You never work directly on main.
1
GitHub remote repo github.com
The central source of truth. Lives on GitHub. Everyone's code eventually lands here on the main branch.
git clone <repo-url>
2
Clone the repo git clone <url>
Downloads the full repository to your local machine. You now have a complete copy including all history and branches. You start on main.
git checkout -b my-feature-branch
3
Create a branch git checkout -b <name>
A safe copy of main for your work. Changes you make here are fully isolated — they won't affect main until you explicitly merge. Name branches descriptively: feat/add-refund-flow.
edit files in VS Code / any editor
4
Make your changes your editor
Edit code, add files, delete things. Everything is local — nothing is shared with the team yet. Use git status to see what's changed at any point.
git add . && git commit -m "description"
5
Stage and commit git add . · git commit
Stage = pack your suitcase. Commit = lock it with a label. A commit is a permanent snapshot with a message. Commit often — small, focused commits are easier to review and reverse.
git push origin <branch-name>
6
Push to GitHub git push origin <branch>
Uploads your branch to the remote repo so your teammates can see it. origin is the default nickname for the repo you cloned from.
open github.com → "Compare & pull request"
7
Open a pull request (PR) github.com
A proposal to merge your branch into main. Write a clear title and description — what changed and why. Assign reviewers. This starts the conversation before anything lands in main.
teammates review, comment, request changes
8
Review and iterate push more commits
Reviewers leave comments on specific lines. You push additional commits to address feedback — they appear in the same PR automatically. Repeat until approved (✅).
click "Merge pull request" on github.com
9
Merge into main github.com merge
Your commits land in main. The branch can now be deleted. Your work is officially part of the codebase and will be deployed in the next release.
git checkout main && git pull origin main
10
Sync your local main git pull origin main
Pull the latest main down to your machine so you're up to date. You're back at the beginning — ready to create the next branch and start again.
The golden rule

Never commit directly to main. Always branch → change → push → PR → review → merge. That single loop is how teams ship safely.

Common situations
Things that trip up newcomers and how to handle them.
Merge conflict
Two people changed the same line. Git pauses and asks you to decide which version to keep. Open the file, look for the <<<<<<< markers, edit to the correct version, then commit.
My branch is behind main
Others merged to main while you were working. Run git pull origin main on your branch to bring in their changes before you push your PR.
Undo the last commit
git reset --soft HEAD~1 removes the commit but keeps your changes staged. Safe to use before you've pushed.
See what changed
git diff shows unstaged changes. git log --oneline lists recent commits. git status is your dashboard.
Key insight

Cursor is VS Code with AI built in. If you already know VS Code, you already know 90% of Cursor. The AI features layer on top — you opt into them when you need them.

Core Cursor features
The six things you'll use every day.
✏️ Inline edit (⌘K)
Select code, press ⌘K (Mac) or Ctrl+K (Win), describe what to change. Cursor shows a diff — accept with Tab, reject with Esc. Works on any selection.
💬 Chat panel (⌘L)
Open a chat sidebar with full codebase context. Ask questions, request explanations, or ask for code that spans multiple files. Reference specific files with @filename.
🤖 Composer / Agent
Give Cursor a multi-step task ("add a dark mode toggle to the settings page") and it plans and executes changes across multiple files autonomously. Review all diffs before accepting.
⚡ Tab autocomplete
Context-aware autocomplete that learns your patterns. Press Tab to accept a suggestion, Esc to dismiss. It predicts multi-line completions, not just the next word.
📎 @ references
In any chat or prompt, use @filename, @folder, @codebase, or @docs to pull specific context into your prompt. Keeps answers accurate and grounded.
📋 .cursorrules
A file in your repo root (like CLAUDE.md) that gives Cursor persistent instructions — your coding standards, stack preferences, naming conventions. Applied to every AI interaction.

Git inside Cursor
The full Git workflow from the previous tab works identically in Cursor — just with extra UI and AI helpers on top.
1
Clone via the terminal panel
Open Cursor, then press ⌃` to open the terminal. Run git clone <url> as normal. Cursor detects the repo and activates Git features automatically.
2
Branch from the status bar
The current branch name shows in the bottom-left status bar. Click it to switch branches or create a new one — no terminal needed. Equivalent to git checkout -b.
3
Stage and commit in the Source Control panel
Press ⌘ Shift G to see all changed files. Click the + icon to stage (equivalent to git add). Type a message and press ⌘ Enter to commit. Or use ⌘K to AI-generate a commit message from your diff.
4
Push and open a PR
Click the sync/push icon in the Source Control panel, or run git push origin <branch> in the terminal. Then open GitHub to raise the PR — same as always.
5
Use AI to write your commit messages
In the Source Control panel, click the sparkle ✨ icon in the commit message field. Cursor reads your diff and generates a conventional commit message — edit before committing.
6
Review PR feedback with AI help
After a reviewer leaves comments, paste the feedback into Cursor's chat panel with ⌘L and ask it to implement the changes. Review the diffs, accept what looks right, push the new commits.
Pro tip

Add a .cursorrules file to every repo with your team's conventions — language version, naming patterns, test requirements. Every engineer on the team gets consistent AI suggestions without having to re-explain the rules each session.
